Chapter E — AVIS Flow: From LLM Command to DirectX Frame

TopicDescription
High‑Level Flow
  1. LLM (FireGem/CYBORG) parses user intent.
  2. CYBORG emits `Cmd` objects.
  3. AVIS verifies CBORD/MERC‑G/CYHY status.
  4. CYBORG routes commands to MERC‑G using DXVector.
  5. MERC‑G issues DirectX calls; CBORD swap chain presents.
Frame Tick 

void MercWar_Tick(float dt)
{
    CYHY_Update();              // health
    CYBORG_PullCommands();      // from LLM / scripts

    if (!avis.dxHealth.deviceLost) {
        CYBORG_ExecuteQueue();  // calls MERC-G
        avis.cbord->dx.swap->Present(1, 0);
    } else {
        CBORD_RebuildDX();      // recovery
    }
}
Vector Out Concept
  • “Vector out” = subsystems never touch raw DX; they use DXVector.
  • DXVector is owned by CBORD, consumed by MERC‑G, referenced by CYBORG/CYHY.
  • AVIS is the registry + health oracle for that vector.
LLM‑Safe Boundary
  • LLM only sees high‑level commands, not COM pointers.
  • Prevents DX misuse; all calls go through MERC‑G wrappers.
  • AVIS + CYHY enforce safety and recovery.
